The features of version control:

  • It allows you to keep a complete record of changes, allowing you to, e.g., revert back to a previous version of your code if things have gone awry.

  • As part of that complete record, it allows you to store messages for every change, so that you can recall the rationale for a change later.

  • With, e.g., Git, you can back your code up in the cloud (e.g., on GitHub), allowing for easy distribution.

Note that here I am talking about â€œcode,â€ but version control can also be useful when you are, for instance, writing reports (like your DAs in 36-401).

Every time you commit your changes, you write a message that clearly indicates the changes that you have made since your last commit.

In Git, two or more people can make independent changes to the same code, or one person can have two or more different code versions being developed in parallel, via branching. Branches

When you first push code to your GitHub repo, it will exist in the so-called main branch. (Up until September 2020, it was called the master branch, and older repos establish before October 2020 still feature a master branch instead of a main one.) You can view this as the â€œdeployable branch,â€ i.e., the branch that contains the code you wish to release to the world.

If you are, e.g., working alone, then a common workflow is to improve your code in your local Git repo, stage, commit, and push to the main branch. In other words, you simply overwrite what is on the main branch as you improve your code.

However, there is another workflow, one that is particularly relevant if you work in teams, and that is to create non-main branches where you can take your (not-yet-ready-to-be-deployed) code in different directions. If you feel that the changes on a branch need to be propagated to the main branch code, you merge your changes into the main branch code via a â€œpull request.â€

Here is a simple series of steps for creating a new branch via GitHub and RStudioâ€™s Git pane:

  1. On GitHub: in your repo, you will see a box above the file list that says â€œBranch: mainâ€ with a downward-pointing triangle. Click on this, and in the window that appears, provide a branch name, such as new_branch. Hit return. GitHub will take you to your new branch. All the code in this branch is the same.
  2. In RStudio: open your project, and go to the Git pane, where you will see the words â€œNew Branchâ€ in the upper-right-hand corner. Click on this, and in the pop-up window that appears, type a branch name. This should be the same name that you just used in the GitHub repo itself.
  3. In the new branch, you can edit code, and the changes will only be seen in the new branch itself. You can stage, commit, and push as before, but your changes will only go to the new branch on GitHub.
